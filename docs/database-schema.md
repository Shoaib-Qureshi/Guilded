# Database Schema

Postgres (via Supabase), accessed through Prisma — see `tech-stack.md` for why. Written in a Prisma-schema-ish shorthand; field names for `applications` intentionally match the existing `lib/validations/onboarding.ts` zod schemas so the current onboarding UI needs no rework later.

## Auth (NextAuth v5 + Prisma Adapter)

Standard adapter tables — not redesigned here, just noted so they're not forgotten in the Prisma schema:

- `Account`, `Session`, `VerificationToken` — as generated by `@auth/prisma-adapter`'s schema snippet.
- `User` (NextAuth's own model) is **extended**, not duplicated — the app's `users` table described below *is* the NextAuth `User` model with extra fields, not a separate table joined to it.

## Why these boundaries

- **`events` vs auth `Session`**: pure naming collision avoidance — Prisma/NextAuth already owns `Session`.
- **Reports point at posts only, not comments, for v1**: the flowchart only draws "view member and post details," not comments; extend later if needed rather than over-modeling now.
- **Messaging is one generic table set for both DMs and moderator contact**: the flowchart's "Submit a message to the moderator" and "View messages / reply via email" are the same underlying primitive (a conversation with 2 participants) as member-to-member DMs — no reason to duplicate schema for it.
- **"Circles" are intentionally not modeled**: the tech doc's notification examples mention "Circle invite" / sub-community discussions, but this isn't in the MVP Scope feature list or either flowchart — see `product-brief.md` and `rules-and-regulations.md` §5. No `circles`/`circle_members` tables yet.
